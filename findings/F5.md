# JWT Tokens Survive Password Changes & Repeated Logouts

**Vulnerability:** Full Account Takeover (Changing passwords has no effect on recovery)

**The Test:**
1. Captured an `A_TOKEN` from a logged-in user.
2. Logged out of the account.
3. Changed the account's password.
4. Logged in and out 3 more times with the new password.
5. Used the **old, initial stolen token** to access the account.

**The Result:**
The old token still worked perfectly. Full dashboard and data accessible.

**Damage:**
- The backend does not tie the JWT to the user's current password hash.
- A stolen token bypasses all security controls—logout, password changes, and multiple new sessions.
- The victim has **zero way** to lock out the attacker. Even changing the password does nothing.

**The Fix:**
When a user changes their password, the backend must **immediately blacklist/revoke all previously issued JWTs for that user ID**. Additionally, every API request should check if the user's password version has changed since the token was issued.
